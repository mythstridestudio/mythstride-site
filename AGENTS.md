# MythStride Website Agent Rules

## Stack

- Next.js
- TypeScript
- Tailwind CSS
- Framer Motion
- GSAP only for scroll/parallax

## Visual Identity

The MythStride website must feel like a premium dark fantasy RPG mobile game landing page.

Use:

- dark fantasy RPG atmosphere
- cinematic composition
- aged gold
- emerald accents
- ember glow
- stone texture
- deep shadows
- mist, depth, ruins, relics
- premium mobile game visual language

## Avoid

Do not use:

- generic SaaS look
- purple/blue AI gradients
- glassmorphism everywhere
- random neon
- boring corporate cards
- overused bento grid
- clean startup template style
- fake dashboards with meaningless data
- excessive animation

## Skills

Always read before making changes:

- skills/frontend-design/SKILL.md
- skills/web-quality/SKILL.md

## Product Direction

MythStride is not a normal running app.

It combines:

- running
- RPG progression
- bosses
- loot
- achievements
- streaks
- guilds
- AI companion / Aethron
- fantasy identity

The website must sell the fantasy:

> Run in the real world. Progress in another.

## Website Goal

The first version is a landing page MVP.

Main goals:

- explain the product quickly
- look premium
- capture interest
- drive app downloads or waitlist signups
- make MythStride feel like a serious product

## Page Structure

Recommended sections:

1. Hero
2. How it works
3. Boss progression
4. Loot and achievements
5. Aethron companion
6. Integrations
7. Player profile / shareable identity
8. Final CTA

## Frontend Rules

Before coding:

1. inspect existing files
2. inspect Tailwind config
3. inspect reusable components
4. reuse tokens and patterns
5. preserve responsiveness
6. preserve accessibility
7. avoid unnecessary dependencies

## Motion Rules

Use Framer Motion for:

- section reveal
- text entrance
- cards
- CTA buttons
- subtle hover states

Use GSAP only for:

- parallax
- scroll-triggered scenes
- layered hero effects

Motion must feel cinematic, not distracting.

## Quality Rules

Before finishing:

- run lint
- run build
- check mobile layout
- remove unused code
- avoid hydration errors
- avoid heavy client components
- keep performance in mind

## Output Expected

When done, report:

- files changed
- visual improvements
- UX improvements
- motion added
- commands tested
- risks or follow-up items
